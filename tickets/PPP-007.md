# Outcome

One turn can safely move from queue to provider result, validation, staged source/database/server, hidden requesting-tab render, exact ACK, durable commit, checkpoint, activation, and broadcast. Every failure preserves the prior active product.

# Scope

- Turn job model and progress events.
- Protocol-versioned authenticated WebSocket subscriptions.
- Requesting-tab identity and pending-stage registry.
- Complete change state machine with per-session commit lock.
- Source materialization, migration staging, server staging, browser staging.
- Exact base/target/transaction ACK validation.
- Journal prepare and durable materialized commit hook.
- History event and checkpoint trigger.
- Other-tab activate/resync behavior.
- Reply and clarify direct completion.

# Acceptance Criteria

- [x] Turn endpoint returns `202` and never blocks on generation.
- [x] Progress visible vocabulary remains four phases.
- [x] Reply/clarify do not change manifest, DB, runtime version, or checkpoints.
- [x] Only requesting tab ACK can authorize commit.
- [x] Wrong/stale/duplicate/late ACK cannot commit.
- [x] Base version is rechecked under commit lock.
- [x] Other tabs never block and receive activate or full resync.
- [x] Server registry, current manifest, and SQLite metadata agree after commit.
- [x] Disconnect/timeout/rejection discards staging and preserves hashes.
- [x] Normal app actions remain available outside the short commit interval.

# Domain/PBT Tests

- PBT-01: random accepted/rejected/restore-like sequences produce unique increasing committed versions.
- PBT-02: every rejection point preserves source and logical DB hashes.
- PBT-08: stale client sequences never commit and receive current manifest.
- Integration tests crash or throw at each pre-commit boundary.
- Multi-tab Playwright proves request-tab authority and follower resync.
- Do not test the exact count or timing of progress events.
