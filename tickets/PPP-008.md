# Outcome

Every successful runtime version has a verified source and SQLite checkpoint. Any checkpoint can be restored through the same browser staging path, and every simulated commit crash converges to a valid base or target state on startup.

# Scope

- Consistent SQLite online backup and gzip integrity verification.
- Checkpoint metadata, source hashes, and logical data hash.
- Checkpoint list and restore endpoints.
- Restore as a new monotonically increasing history event.
- Preserve all future checkpoints.
- Codex thread reset after restore.
- Prepared journal before backup and staged after-state references.
- Startup recovery finalize/rollback matrix.
- Recovery registry reactivation before readiness.

# Acceptance Criteria

- [x] Version zero blank session has a valid checkpoint.
- [x] Successful changes create one checkpoint after durable commit.
- [x] Reply/clarify/rejected changes create none.
- [x] Restore validates source, gzip, database integrity, and logical hash before staging.
- [x] Restore rewinds source and user data together but receives a new version/event.
- [x] Later checkpoints remain restorable.
- [x] Restore clears Codex thread ID.
- [x] Every journal state recovers idempotently before readiness.
- [x] Corrupt checkpoint leaves current product unchanged.

# Domain/PBT Tests

- PBT-03: restore(A), arbitrary valid changes, restore(A) equals A source and logical data hashes.
- PBT-04: execute startup recovery twice at every injected crash point and get the same valid state.
- Random corruption of manifest/source/gzip/SQLite is rejected.
- Snapshot under concurrent action writes is internally consistent.

# Out of Scope

- Automatic checkpoint deletion or compaction.
- Git revert or branch creation.
- Distributed database recovery.
