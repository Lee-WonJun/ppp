# Outcome

A real Codex OAuth session can evolve one running product through multiple
client-only and server-backed turns without refresh. The kernel classifies the
actual returned file set, stages only the affected generated runtime surfaces,
feeds bounded failures back to the same Codex thread, and records enough
non-secret evidence to prove what ran.

# Scope

- Host-derived `:client-only` and `:server-data` change-impact classification.
- Client-only changes clone the atomic SQLite snapshot but reuse the already
  validated generated server runtime instead of evaluating server SCI or domain
  tests again.
- Server/shared/test/migration changes stage fresh server SCI and execute
  rollback-only generated domain tests.
- Provider guidance that preserves unrelated features and keeps local games,
  timers, keyboard input, and Canvas entirely in the sandboxed browser runtime.
- An explicit `bb eval-evolution` OAuth gate that uses one fresh session and one
  resumed Codex thread for a realistic multi-turn evolution.
- Sanitized report evidence for changed surfaces, repair count, browser outcome,
  SQLite persistence, server rule replacement, and disposal of a replaced game.

# Acceptance Criteria

- [x] A client/CSS-only change commits with history validation
  `:server :not-applicable`, leaves generated server/shared/test source and the
  logical SQLite content unchanged, and keeps the active server actions usable.
- [x] A server/shared/test or migration change receives a fresh server stage,
  generated domain-test stage, and atomic SQLite/client activation.
- [x] Every successful change records host-derived impact, generation-attempt
  count, and semantic stage applicability without exposing source in logs.
- [x] Browser/source/server rejection can request up to two corrected attempts
  in the same Codex thread and the final history records the successful attempt
  count.
- [x] `bb eval-evolution` uses the real OAuth Codex provider, not the fake
  provider, and evolves one fresh session through visual, game, persisted
  ranking, ranking-rule, and game-replacement turns.
- [x] The live loop proves timer-driven and keyboard game behavior, SQLite
  persistence across reload, a changed server ranking result, preservation of
  unrelated ranking data during game replacement, and removal of the old game.
- [x] The loop emits a bounded report that fails closed when a requested outcome,
  runtime-surface boundary, stage, or history event is missing.

# Domain/PBT Tests

- Generate allowed change path sets and prove impact classification is
  deterministic and cannot label server/shared/test/migration work client-only.
- Client-only commit preserves logical SQLite hash and action behavior while
  advancing the runtime/registry version exactly once.
- Server-data changes cannot skip generated tests.
- Evolution report records fail when client-only events touch server surfaces or
  when server-data events omit server/domain-test validation.

# Evidence

- The impact classifier property ran 1,000 generated path sets (seed `14001`),
  with focused coordinator, server-runtime reuse, provider-guidance, and
  fail-closed report tests included in the JVM gate.
- `artifacts/evolution-eval/20260715-234942/report.edn` records a real OAuth
  six-turn run in one session and one resumed Codex thread: six of six scenarios
  passed, EVOLVE-04 repaired itself on attempt two, browser-only turns skipped
  generated server staging, and server-backed turns ran server/domain stages.
- The browser observations prove dark and floating presentation, timer and
  keyboard Tetris behavior, SQLite persistence after reload, a server ranking
  change from 10 to 20, Tetris removal, Gomoku interaction, and preservation of
  the existing ranking row.
- `bb verify` passes 132 JVM tests/750 assertions, 17 CLJS tests/73 assertions,
  eight semantic browser paths, the amd64 non-root/read-only Docker smoke, and a
  169-file secret scan.
- The owner-facing `http://localhost:8787` Chromium smoke passes blank-canvas
  access, sidebar opening, sequential composer input with retained focus, new
  session creation, and reload restoration. Its two temporary sessions were
  deleted afterward; the restarted server reports zero sessions and zero
  registered runtimes while preserving OAuth readiness.

# Out of Scope

- Running live OAuth evaluation in `bb verify` or CI.
- Letting generated client code bypass the opaque-origin sandbox.
- Avoiding the kernel request/history/checkpoint path for conversational changes.
- Partial source patches or a model-declared impact flag.
- Compatibility branches or a capability-version increment before first release.
