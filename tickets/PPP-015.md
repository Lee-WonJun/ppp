# Outcome

The conversation composer behaves like a chat surface: pressing Enter sends a
nonblank draft immediately, while Shift+Enter inserts a newline for intentional
multiline prompts. Keyboard submission follows the same guarded turn path as the
Send control and does not lose focus or submit twice.

# Scope

- Define the composer keyboard contract in `DESIGN.md`.
- Route unmodified Enter through the existing send-turn function.
- Preserve Shift+Enter as the multiline escape hatch.
- Keep the current busy, blank-draft, failure, and draft-preservation rules.
- Cover the behavior at the CLJS boundary and in a compiled Chromium runtime.

# Acceptance Criteria

- [x] Enter on a nonblank idle composer prevents a newline and submits exactly
  one turn through the existing transport path.
- [x] Shift+Enter inserts a newline and does not submit.
- [x] Enter on a blank or busy composer does not submit.
- [x] Keyboard submission preserves the existing focus and recoverable-failure
  behavior.
- [x] The compiled browser behavior passes on an isolated port before paired
  host/frame bundles are promoted to port 8787.
- [x] The owner-facing port 8787 passes the original semantic typing and Enter
  submission path after promotion.

# Domain/PBT Tests

- Unit examples cover Enter, Shift+Enter, blank, and busy decisions as a public
  composer interaction contract.
- Browser verification uses semantic roles and observable conversation/turn
  state; it does not depend on CSS classes, DOM nesting, or exact prose.

  composer recognition. The complete CLJS gate passes 21 tests/89 assertions

- A release build in `/tmp/ppp-015.wvVSO4` passes all 11 Chromium paths. The
  composer path passes in three fresh contexts with fresh sessions; one delays

- The paired host/frame release hashes were promoted together while port 8787

- Final regression passes 132 JVM tests/750 assertions, 21 CLJS tests/89

# Out of Scope

- Removing multiline prompts.
