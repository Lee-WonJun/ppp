# Outcome

PPP is a locally complete, reproducible hackathon release candidate rather than
an uncommitted working directory. Product, security, demo, and submission claims
match the current runtime; only remote publication, external deployment, video
upload, and Devpost submission remain behind explicit owner approval.

# Scope

- Reconcile README, demo, Devpost, rubric, traceability, and release wording
  with opaque-frame JavaScript interop, selective runtime staging, automatic
  repair, Enter-to-send, and the current evidence counts.
- Make the exact packaged demo use the same Enter chat interaction as the
  owner-facing product and rerun it three consecutive times on fresh volumes.
- Track only bounded sanitized reports and deliberate screenshots/subtitles;
  exclude raw observations, provider output, sessions, generated bundles, and
  transient Playwright artifacts from the repository baseline.
- Run the complete fake-provider release contract in a clean temporary copy so
  port 8787 never serves a partial browser build.
- Verify the promoted stable surface, reset pre-release sessions, preserve
  OAuth/kernel state, and create the first intentional local Git commit.
- Produce a local release record that names every remaining owner-authorized
  external action without claiming it was performed.

# Acceptance Criteria

- [ ] Documentation contains no claim that browser JavaScript interop is
  forbidden and describes Enter/Shift+Enter plus selective runtime staging.
- [ ] Packaged demo sends conversation turns with Enter and passes 3/3 from
  fresh data/Codex-home volumes without terminal intervention.
- [ ] `bb verify` passes from a clean temporary repository copy after PPP-015,
  including release builds, semantic browser tests, Docker smoke, and secret
  scan.
- [ ] Repository candidate contains no raw live observations, provider JSONL,
  session databases, OAuth material, generated browser bundles, or transient
  browser reports; referenced sanitized evidence remains available.
- [ ] Port 8787 runs the paired verified browser bundles, reports provider ready,
  and has zero development sessions and zero in-memory runtimes at handoff.
- [ ] All local source, documentation, tests, and allowed evidence are captured
  in an intentional local commit; no push, deployment, upload, or submission is
  performed.
- [ ] Devpost and release checklists distinguish completed local work from the
  exact external actions awaiting owner approval.

# Domain/PBT Tests

- Existing PBT-01 through PBT-08 remain the release-domain contract.
- Repository hygiene checks enumerate the actual candidate file set and fail on
  forbidden names, credential patterns, raw observation files, or generated
  runtime data.
- Packaged and stable browser verification uses semantic roles and persistent
  outcomes; it does not assert incidental copy, classes, or DOM nesting.
