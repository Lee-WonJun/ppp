# Outcome

A valid saved or newly created product finishes hidden browser staging even when
the browser suppresses `requestAnimationFrame` for the invisible sandbox frame.
The host still rejects a generated surface that throws or never reaches a real
React DOM commit, and the previously saved product remains unchanged.

# Scope

- Replace animation-frame-based staging completion with an explicit DOM commit
  signal owned by the immutable frame shell.
- Coalesce approved page-state and host-bridge changes into a microtask render
  flush so an active opaque frame remains interactive without animation
  scheduling.
- Preserve the existing load deadline, render deadline, error boundary, exact
  version matching, and opaque-origin frame boundary.
- Add a browser regression that disables animation frames inside every sandbox
  frame and exercises initial version-zero load plus a fresh session.
- Run the owner path for more than the former ten-second boundary in three fresh
  browser contexts, including one delayed frame-bundle run.
- Promote the verified host and frame bundles together to port 8787, reset
  pre-release sessions, and restart the JVM.

# Acceptance Criteria

- [x] Initial version-zero staging succeeds when sandbox
  `requestAnimationFrame` never calls its callback.
- [x] Creating a fresh session under the same condition activates version zero
  without showing the ten-second drawing error.
- [x] A generated render exception still rejects staging with a bounded,
  stage-specific reason and does not replace the active product.
- [x] The stage-success signal corresponds to an actual React DOM commit, not
  only source evaluation or a larger timeout.
- [x] After activation with animation frames still suppressed, the immutable
  handle closes and reopens the generated conversation panel and approved page
  state continues to trigger drawing.
- [x] Focused browser verification passes in three fresh contexts; each waits
  beyond ten seconds, and one also delays the frame bundle by 6.5 seconds.
- [x] The complete `bb verify` gate passes from the promoted source.
- [x] Stable port 8787 serves only the paired verified bundles and starts with
  zero development sessions and zero in-memory runtimes.
- [x] The specification and traceability ledger describe the corrected commit
  contract and retain the original failure evidence.

# Domain/PBT Tests

- Browser property: staging liveness does not depend on animation scheduling in
  a hidden or background sandbox.
- Safety property: no commit signal is emitted before the generated page and
  sidebar share one successful DOM commit.
- Existing rejected-change preservation, stale-version, Safe Mode, timer,
  keyboard, Canvas, WebAssembly, SQLite action, and opaque-parent tests remain
  unchanged and passing.

# Evidence

- Confirmed pre-fix reproduction: overriding sandbox `requestAnimationFrame`
  with a non-calling function leaves saved version 0 inactive and produces
  `:runtime/client-frame-render-timeout` after the ten-second render deadline.
- Focused post-fix regression: three of three fresh Chromium contexts passed
  with animation frames continuously suppressed; each survived 10.5 seconds,
  closed and reopened the sidebar, and the delayed case also held the frame
  bundle for 6.5 seconds.
- Clean isolated `bb verify`: 132 JVM tests/750 assertions, 21 CLJS tests/89
  assertions, release host/frame builds without warnings, Chromium 14/14,
  Docker smoke, and secret scan of 156 candidate files passed.
- Stable `localhost:8787`: three additional fresh contexts passed with observed
  new-session times of 11.2, 11.3, and 17.7 seconds; all retained version 0,
  one active frame, a visible sidebar, and no runtime diagnostic. The final
  ordinary browser run remained usable after 12 seconds with no console error.
- Paired promoted bundle hashes: host
  `fdedc4b92a0f039269ac447a474d5487ece8f9ff153e3bfeb5cc5ed9f814f0df`,
  frame
  `46e108fed93af583daf725e22f740c66d17abb72c614a2877fc72c679a3729b3`.
- Final readiness: Codex OAuth ready, zero sessions, zero runtime registries,
  zero connections, and zero pending stages after deleting five validation
  sessions and restarting the JVM.

# Out of Scope

- Increasing the render timeout as a workaround.
- Removing hidden staging or weakening render-error rejection.
- Changing the opaque-origin sandbox or exposing parent authority.
- Migrating existing development sessions or incrementing
  `capability-version` before the first public release.
- External push, deployment, image publication, or Devpost submission.
