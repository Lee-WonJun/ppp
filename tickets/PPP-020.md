# Outcome

Generated products can implement real signup, login, logout, authenticated
reload, and user-owned server behavior without receiving host credentials,
cookies, raw response control, filesystem access, or another session's data.
The same work defines the general playground rule: normal product behavior is
allowed inside session-owned virtual resources, while effects that could cross
the session sandbox remain mediated by typed Kernel capabilities.

# Scope

- Reconcile PRD, implementation, security, design, rubric, and traceability
  around the distinction between PPP workspace access and accounts belonging
  to a generated product.
- Record a capability case matrix for browser behavior, relational data,
  identity, authorization, public network calls, connectors, binary assets,
  realtime events, background work, inbound endpoints, and forbidden host
  authority.
- Add a Kernel-owned product identity service backed by reserved tables in each
  session SQLite database.
- Hash passwords with Argon2id, generate opaque session tokens with
  `SecureRandom`, store only keyed token digests, and apply bounded login
  throttling.
- Carry an authenticated product-user context into generated server actions
  without changing the action payload contract.
- Expose typed server capabilities for registration, login, logout, current
  user, required user, password change, and account deletion.
- Translate successful login/logout effects into session-scoped HttpOnly,
  SameSite cookies in the authenticated parent action transport. Never expose
  the cookie or token to generated client or server source.
- Preserve accounts across normal runtime changes and reloads. Restoring a
  checkpoint restores account records but revokes every live product login.
- Update Codex guidance and the deterministic fake provider so product-account
  requests produce a working full-stack change rather than an access-gate
  refusal.
- Add focused unit, property, integration, generated-domain, and real Chromium
  coverage.

# Acceptance Criteria

- [x] The docs define the sandbox by owned resources and forbidden effects,
  not by an arbitrary list of app categories.
- [x] PPP platform identity remains out of scope, while generated product
  identity is explicitly supported.
- [x] A generated signup action creates one account, a product profile, and an
  authenticated session atomically.
- [x] Passwords are never stored or logged in plaintext; stored credentials use
  bounded Argon2id parameters and a unique salt.
- [x] Login returns the same public failure for an unknown identifier and a
  wrong password, and repeated failures are throttled.
- [x] Product session tokens are opaque, high entropy, stored only as keyed
  digests, scoped to one PPP session, HttpOnly, SameSite=Strict, and Secure
  outside development.
- [x] Generated source cannot read tokens, password hashes, reserved auth
  tables, PPP access state, or another product session.
- [x] Authenticated actions receive only bounded public user claims through
  `current-user`/`require-user!`; product roles and profiles remain ordinary
  generated-domain data.
- [x] Signup and login survive a browser reload; logout and password change
  revoke the previous session.
- [x] Checkpoint restore preserves the selected account state but revokes all
  pre-restore login tokens.
- [x] A deterministic fake-provider product supports signup, login, protected
  behavior, logout, and clear recoverable failures.
- [x] The exact owner-reported request, “implement login and signup for this
  game,” returns and applies a change instead of a refusal in the live provider
  evaluation.
- [x] Three fresh browser contexts prove signup/login/reload/logout and
  cross-context isolation. One run delays the sandbox frame beyond five
  seconds.
- [x] `bb verify` passes from an isolated candidate copy before paired browser
  bundles are promoted to the stable development surface.
- [x] Because this changes the pre-release runtime contract, capability version
  remains `1`, development sessions are reset, and the stable JVM is restarted
  with an empty in-memory registry.

# Domain/PBT Tests

- For arbitrary distinct session UUIDs, a product-auth cookie name, path, and
  token from one session never authenticate another.
- For arbitrary supported Unicode identifiers, normalization is deterministic,
  bounded, and case-insensitive without permitting control characters.
- Registering the same normalized identifier at most once preserves the first
  account and never returns a credential hash.
- Unknown-user and wrong-password login share one public error category.
- Expired, revoked, password-version-stale, and post-restore tokens never
  produce a current user.
- Failed signup/profile creation and failed authenticated business actions roll
  back both auth and ordinary product data.
- Auth response effects never enter JSON bodies, history, generated state, or
  application logs.

# Out of Scope

- PPP's own public signup, workspace tenancy, billing, invitations, or hosted
  account recovery.
- Giving generated code raw `Set-Cookie`, arbitrary response headers, PPP
  access cookies, CSRF values, host filesystem paths, shell, processes, or
  Kernel tables.
- Shipping a built-in email/SMS provider, social OAuth application, payment
  processor, or secret-management UI in this ticket. Those product flows use
  typed/named connectors when configured and remain represented in the
  capability matrix.
- External push, deployment, image publication, or Devpost submission.
