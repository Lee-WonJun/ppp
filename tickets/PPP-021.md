# Outcome

Generated products can compose durable assets, live multi-tab behavior,
scheduled/background work, public APIs/webhooks, and full-text/vector search
without receiving host filesystem, thread, socket, listener, credential,
Kernel-table, or cross-session authority. These capabilities share the same
session ownership, transaction, quota, checkpoint, restore, failure, fake/live
provider, and browser evidence contracts as product data and identity.

# Scope

- Add one Kernel-owned resource service for reserved SQLite blob, job, and
  search state.
- Add bounded server capabilities for blob CRUD, product event publication,
  named job registration/scheduling/status/cancellation, named ingress
  registration, and text/vector search.
- Add rollback-only generated-test helpers for registered jobs and ingress.
- Add a Kernel scheduler with durable lease, retry, idempotency, cancellation,
  crash recovery, current-runtime lookup, and post-commit effect dispatch.
- Add a public ingress route with UUID/route containment, method/body/header
  policy, rate limiting, optional developer-owned HMAC verifier alias, safe
  status/body responses, and no PPP/product-cookie authority.
- Broadcast bounded product events to current exact-session/runtime WebSocket
  subscribers and forward them only to the active opaque frame.
- Add generated client event registration under the existing execution budget
  and frame disposal boundary.
- Keep blobs and search documents in snapshots. On restore, cancel every
  restored pending/running job and discard ephemeral events while restoring
  generated ingress definitions from source.
- Extend provider context, packaged Skill, fake provider, resource catalog,
  evolution evaluator, docs, and release evidence.

# Acceptance Criteria

- [x] A generated action stores, replaces, lists, reads, and deletes an exact
  binary object up to 4 MiB without a path capability; metadata includes
  bounded name/type/size/SHA-256 and checkpoint restore recovers exact bytes.
- [x] Blob count/size/database quota failures are pre-mutation and preserve the
  prior object and active product.
- [x] A committed action, successful job, and successful ingress can publish a
  bounded event to all tabs on the exact session/runtime.
- [x] Rolled-back, failed, stale-runtime, restored, and cross-session work emits
  no event; reconnect reconstructs from SQLite rather than event replay.
- [x] Generated source registers named job handlers and schedules durable jobs
  with bounded payload/delay/attempts and optional idempotency key.
- [x] The Kernel runner executes due jobs without another user action, commits
  their SQLite mutation and effects atomically, retries with bounded backoff,
  recovers expired leases, and records terminal status.
- [x] Cancellation prevents pending execution, one handler/idempotency key
  creates at most one logical job, and checkpoint restore cancels restored
  pending/running work before activation.
- [x] Generated source registers named public ingress handlers; exact session
  and route selection, bounded methods/body/safe metadata, per-route/address
  rate limit, and JSON status/body contract are enforced before invocation.
- [x] Optional ingress HMAC verification uses a developer-owned alias and
  constant-time comparison without exposing secret values or environment names
  to provider, source, request, response, history, or logs.
- [x] Search upsert/delete/query supports Unicode FTS relevance and optional
  2-1536 finite numeric vectors with bounded candidates/results and
  deterministic ties inside one session only.
- [x] Generated SQL cannot enumerate or modify `_ppp_blobs`, `_ppp_jobs`, or
  `_ppp_search_*`; generated source never sees a host path, scheduler/thread,
  socket/listener, verifier secret, another runtime registry, or another
  session's resources.
- [x] Fake-provider browser flow proves upload/reload, search, background
  completion, second-tab event, public ingress, checkpoint/restore policy, and
  failure reasons using real SQLite and compiled host/frame bundles.
- [x] The real Codex evolution extends the existing game/account product with
  the complete resource workbench in one resumed thread and preserves every
  prior unrelated feature. Any committed semantic miss receives bounded
  browser feedback in that same thread; repair versions remain visible and
  must all pass staging and report coverage.
- [x] `bb verify` passes in an isolated candidate, both browser bundles are
  promoted together, capability version remains `1`, dev sessions reset, and
  the stable JVM restarts with zero stored/registered runtimes.
- [x] PRD, SPEC, SECURITY, DESIGN, RUBRIC, TODOS, TRACEABILITY, provider Skill,
  and evidence agree that the known ordinary session resource ledger has no
  remaining implementation gap.

# Domain/PBT Tests

- PBT-11: arbitrary accepted bytes round-trip with identical SHA-256, size,
  metadata, and restored bytes; over-limit and path-shaped values preserve the
  previous resource hash.
- PBT-12: arbitrary commit/failure/session/runtime/subscriber sequences deliver
  each accepted event only to the exact eligible tabs and never from rollback.
- PBT-13: arbitrary schedule/claim/fail/retry/cancel/crash/restore sequences
  preserve idempotency, attempt bounds, terminality, and post-restore
  non-execution.
- PBT-14: arbitrary ingress method/route/session/body/header/rate/signature
  combinations cannot bypass ownership, bounds, verifier, or handler lookup.
- PBT-15: arbitrary Unicode documents, bounded vectors, queries, and session
  pairs produce bounded deterministic results only from the selected session.
- Generated domain tests exercise one action, job, ingress, blob, search
  mutation, and read reconstruction inside rollback-only staging.

# Out of Scope

- Generated shell, filesystem paths, JVM/process/thread ownership, arbitrary
  socket servers, dynamic dependencies, PPP credentials/Kernel tables, and
  cross-session access. These are permanent sandbox escapes, not follow-up
  product work.
- PPP hosted signup/tenancy, multi-node distributed scheduling, exactly-once
  delivery to external systems, user-managed secret UI, email/SMS/social OAuth
  provider purchase, or external deployment. These change deployment and
  organizational authority rather than the completeness of one session
  sandbox.
- Push, PR, image publication, VPS/Coolify deployment, video upload, or Devpost
  submission without explicit owner approval.
