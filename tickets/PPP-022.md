# Outcome

Hackathon judges can open one hosted PPP instance, enter one owner-provided
shared password, see every existing project in the shared `local` workspace,
open or create a named project, and use the complete live product runtime. The
instance protects the owner's Codex OAuth capacity with a persistent rolling
global limit without inventing judge accounts, ownership, private copies, or
per-person quotas.

# Scope

- Replace the production access-fragment entry path with an origin-checked
  shared-password form backed by the existing signed `ppp_access` cookie.
- Keep fragment exchange available only when explicitly enabled for local
  development and deterministic tests.
- Add logout, cookie expiry, and bounded per-remote-address failed-login
  throttling without storing or logging submitted passwords.
- Add an immutable Projects surface that lists every session in workspace
  `local`, opens deep-linked existing projects, and creates a named blank
  project without automatically opening or creating the first session.
- Keep every authorized browser on the same shared project list. Do not add a
  user, judge, owner, membership, or per-project authorization model.
- Add Projects navigation to the default conversation and Safe Mode recovery
  surfaces while preserving the generated sidebar's freedom to redesign.
- Persist actual Codex provider-start timestamps under `data/kernel` and
  enforce at most 100 starts in any rolling 60-minute window by default.
- Count the initial generation and every repair attempt once immediately
  before provider invocation. Do not count fake-provider work, queue waiting,
  replies after completion, generated actions, restores, or product jobs.
- Reject new AI turns when capacity is exhausted with HTTP 429 where admission
  can fail synchronously, a bounded `Retry-After`, and the same observable
  failure over the asynchronous turn channel if an already queued job reaches
  the boundary later.
- Keep project opening, generated actions, SQLite persistence, history,
  checkpoints, restores, Safe Mode, and logout usable while AI changes are
  temporarily unavailable.
- Add owner-only local commands to inspect and reset the provider-start ledger.
- Update deployment and Devpost instructions for one replica, two persistent
  volumes, and private judge-field password delivery.

# Acceptance Criteria

- [x] Production starts with a semantic Password form and does not accept an
  access fragment unless `PPP_FRAGMENT_ACCESS_ENABLED=true` is explicitly set.
- [x] A correct shared password sets the signed HttpOnly, Secure,
  SameSite=Strict cookie; an incorrect password returns one generic outcome and
  never enters logs, history, or browser storage.
- [x] Ten failed attempts from one kernel-observed remote address within ten
  minutes are throttled while a different address and a later rolling window
  remain usable.
- [x] Logout clears the access cookie, closes the live connection, disposes
  generated frames, and returns to the Password form without deleting data.
- [x] Successful login opens Projects, not an automatically selected or
  automatically created session, and lists all existing `local` sessions by
  bounded title and last-updated time.
- [x] Opening a project preserves the existing `?session=<uuid>` deep link;
  opening a valid deep link after login goes directly to that project, while a
  missing/stale deep link returns to Projects without opening another project.
- [x] New project accepts a nonblank title of at most 80 characters, creates a
  complete version-zero source/SQLite/checkpoint state, then opens its literal
  blank canvas and immutable handle.
- [x] Every browser authenticated with the shared password observes the same
  project list and can open/change the same project; no judge identity or
  ownership field exists.
- [x] Default conversation and Safe Mode provide a semantic All projects
  action; custom generated sidebars cannot remove Safe Mode recovery access.
- [x] The real Codex provider allows at most 100 actual starts in every rolling
  60-minute window by default, counts repair attempts, and persists accepted
  starts across JVM restart with an atomic no-symlink ledger.
- [x] Fake-provider verification and CI consume no provider capacity. Owner
  status/reset commands operate only on the configured data directory and
  print no credential, prompt, source, or session data.
- [x] Capacity exhaustion has a stable internal code, public 429 plus bounded
  `Retry-After` when known before queue admission, nontechnical UI copy, and no
  source/SQLite/version/checkpoint mutation.
- [x] Existing projects, generated actions, reload persistence, checkpoints,
  explicit restores, Safe Mode, and logout pass while provider capacity is
  exhausted.
- [x] Three fresh production-configured browser contexts pass
  login -> Projects -> create/open -> conversation -> logout, including a
  second browser seeing the first browser's project and a restart preserving
  both projects and provider usage.
- [x] Focused JVM/CLJS/Playwright/Docker gates and the isolated `bb verify`
  release contract pass before paired browser bundles reach port `8787`.
- [x] PRD, SPEC, SECURITY, DESIGN, RUBRIC, TRACEABILITY, README, deployment,
  and Devpost instructions describe the same shared-workspace and rate policy.

# Domain/PBT Tests

- PBT-16: for arbitrary provider-attempt timestamps around the rolling-window
  boundary, at most the configured number are admitted, pruned starts become
  available exactly after the window, and restart/reload yields the same
  decision.
- PBT-17: arbitrary accepted/rejected login sequences remain isolated by the
  kernel-observed remote-address bucket, never reveal whether a password is
  close/correct through public failure categories, and recover after the
  rolling throttle window.
- Session title normalization accepts only bounded nonblank strings and never
  changes the complete version-zero product invariant.
- Capacity rejection preserves current source manifest, logical SQLite hash,
  runtime version, checkpoints, and action availability.

# Out of Scope

- PPP user signup, judge identities, project ownership, private copies,
  organizations, invitations, per-user quotas, billing, audit attribution, or
  account recovery.
- Project rename/delete, templates, demo-session protection, judge-specific
  seed state, or automatic cleanup.
- Public password disclosure. The hosted URL and shared password belong only
  in Devpost's private judge instructions.
- Multi-replica rate coordination, distributed queues, public SaaS metering,
  Responses API billing, or service-account credentials.
- Push, PR, image publication, VPS/Coolify deployment, DNS changes, video
  upload, or Devpost submission without explicit owner approval.
