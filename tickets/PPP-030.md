# Outcome

Readers can separately understand PPP's product vision, the current hackathon
implementation, why it was implemented with in-process SCI, its limitations,
and the intended Codex-operated nREPL/CLJS-REPL workspace architecture. The
current implementation is described accurately as SCI-evaluated transactional
hot swap, not falsely as a direct nREPL connection or in-place REPL mutation.

# Scope

- Add one stable five-part positioning model: Vision, Current Implementation,
  Why This Implementation, Current Limits, and Wannabe Architecture.
- Define precise evaluation terminology for the current and intended runtimes.
- Reconcile PRD, specification, security, thesis, README, long-term work, and
  Devpost copy.
- Replace the old claim that shell/filesystem/dependencies are permanently
  outside the product with a profile-specific boundary.
- Keep all current hackathon acceptance and security requirements unchanged.
- Synchronize the reversible public Devpost description only.

# Acceptance Criteria

- [x] Current code is called an SCI-evaluated transactional hot swap, never a
  current direct nREPL integration or in-place live-context mutation.
- [x] The public POC is identified as one shared server spending the owner's
  Codex OAuth capacity, which explains its in-process restrictions.
- [x] The target architecture gives Codex broad nREPL, browser REPL, source,
  shell, filesystem, and dependency authority only inside a disposable
  per-workspace sandbox.
- [x] Kernel, credentials, other workspaces, host resources, and control-plane
  authority remain outside the target sandbox.
- [x] REPL-first exploration reconciles to source and data at checkpoint and
  handoff boundaries.
- [x] Upper and lower source-of-truth documents no longer conflict on whether
  workspace-local shell/dependencies are a permanent product gap.
- [x] Public Devpost copy agrees while video and submission stay untouched.
- [x] Lint, formatting, diff, reference, and staged secret checks pass.
