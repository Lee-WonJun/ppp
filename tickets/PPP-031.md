# Outcome

Codex connects to a long-lived workspace nREPL and changes the running product
server by defining and redefining actual JVM Clojure Vars used by the live
action router. The same change turn evaluates
client forms in the currently connected browser runtime. Codex inspects,
evaluates, observes, receives bounded failure feedback, and repairs in the same
live nREPL session. Forms change the workspace immediately; only the accepted
state is reconciled to source, tests, SQLite, semantic history, and a recoverable
checkpoint. An exhausted turn reconstructs the last durable checkpoint.

# Scope

- Make Workspace REPL the primary product runtime contract and retain the
  Shared Public POC hot-swap profile only as an explicitly bounded fallback.
- Add a loopback/internal nREPL service with one long-lived logical session per
  PPP project and no public listener.
- Give Codex a project-scoped REPL client Skill/tool in the Workspace REPL
  profile; it must inspect and evaluate the running product before returning an
  accepted change.
- Evaluate incremental server forms against the active workspace runtime,
  retaining real JVM namespace Vars and state across repair attempts. The
  action router retains a Var so redefining it changes the next invocation.
- Evaluate incremental CLJS forms in the requesting browser's long-lived
  sandbox runtime and preserve local state.
- Invoke server actions and browser renders as observable checks before
  acceptance.
- Reconcile the accepted runtime branch to complete source, domain tests,
  migrations, SQLite, semantic history, and checkpoint.
- Restore by rebuilding both live REPLs from the selected checkpoint and
  starting a fresh Codex thread/REPL branch.
- Preserve Safe Mode, failure observability, stale-tab rejection, quotas, and
  source-plus-data crash recovery.
- Reconcile PRD, specification, security, thesis, README, rubric, Devpost, and
  traceability with the implemented mechanism.

# Acceptance Criteria

- [x] Codex uses nREPL to inspect and incrementally evaluate a running project
  server; a full-file proposal alone cannot satisfy the Workspace REPL profile.
- [x] Successive forms and turns share the project nREPL namespace and active
  runtime instead of creating a fresh evaluator for every proposal; redefining
  a registered Var changes the next real action without source staging.
- [x] A rejected form returns its exact bounded reason and the same live nREPL
  remains available for repair; if the turn exhausts repair, the last durable
  server and browser checkpoint is reconstructed.
- [x] Browser form evaluation changes the open product without page refresh,
  preserves serializable UI state, and returns render/console/action failure
  evidence to the same turn.
- [x] A server-business-rule redefinition changes the next real HTTP action and
  can use the existing per-project SQLite data.
- [x] Acceptance materializes reproducible CLJ/CLJS/CLJC/CSS/SQL/tests and a
  semantic history record containing intent, evaluated forms, observations,
  validation, and checkpoint identity.
- [ ] Restore reconstructs the same server/client behavior and data and prevents
  future REPL context from contaminating the restored branch.
- [x] nREPL is loopback/internal only; the public HTTP/WS surface cannot submit
  arbitrary evaluator messages.
- [x] The Shared Public POC fallback is accurately named transactional hot swap,
  not REPL-driven development.
- [x] Real Chromium proves incremental client state, timer/keyboard/Canvas/WASM,
  a persistent server action, failure repair, restore, and Safe Mode in three
  fresh contexts.
- [ ] `bb verify`, a dedicated REPL evolution gate, packaged smoke, and secret
  scan pass.

# Domain/PBT Tests

- REPL session identity and namespace state persist across at least 1,000
  generated evaluation sequences.
- Failed or abandoned turns cannot change durable source, SQLite hash, or
  checkpoint list, and terminal rollback reconstructs their active behavior.
- Commit reconciles exactly one live session and stale runtime versions cannot
  commit.
- `restore(A) -> arbitrary REPL evaluations -> restore(A)` reproduces source,
  SQLite content, active definitions, and browser manifest.
- nREPL messages cannot select another project, access a missing branch, exceed
  code/result bounds, or reach the public listener.
- Browser evaluation acknowledgements are bound to request tab, project,
  branch, and base runtime version.

# Out of Scope

- Public unauthenticated nREPL or arbitrary evaluator endpoint.
- Claiming plain same-process nREPL is a multi-tenant isolation boundary.
- Git/PR source promotion automation.
- Public whiteboard collaboration, CRDTs, or branch-merge UI.
- External deployment, credential publication, video upload, or Devpost
  submission without separate owner approval.
