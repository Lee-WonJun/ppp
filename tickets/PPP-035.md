# Outcome

The product contract treats application auth, data, and diagnostics as
workspace-owned authority while keeping only the PPP Control Plane, provider
credentials, host, and other workspaces outside the model boundary. The final
submission film uses a new real public-server Codex run to show a
visible dark-theme transformation, generated-product signup and sign-in, and a
browser game growing into a platform. Prompt entry looks like a person typing
and pressing Enter, results stay in view, and long provider waits are honestly
but aggressively accelerated.

# Scope

- Replace the obsolete public final-video story and recording assertions.
- Correct the public-POC versus Workspace Capsule boundary across authoritative
  documents and provider context: typed auth is a shared-host implementation
  choice, not the intended permanent product boundary.
- Make failed targeted server actions observable as bounded reactive product
  state so generated account UI cannot remain optimistically signed in or stuck
  on loading copy after a rejected request.
- Type each visible prompt quickly character by character, scroll it into view,
  press Enter to submit, and scroll the resulting conversation outcome into
  view before showing the generated product.
- Record a fresh public project using the configured real OAuth Codex provider.
- Show a dark-theme change and a generated signup/login flow backed by the
  product-auth server boundary.
- Keep browser-owned timer/keyboard game behavior and evolve the product into a
  multi-game library without losing account behavior.
- Compress genuine provider waits substantially more than the former 6x-8x
  treatment while retaining a visible truthful acceleration label and direct
  continuity into each real outcome.
- State that Codex with GPT-5.6 Terra at medium reasoning supported planning,
  implementation, the presentation deck and film, and is the live agent behind
  the current public test server.
- Rebuild and verify the English-narrated, English-captioned final film.
- Push and deploy the verified host/runtime revision, validate the judge origin,
  publish the final video, and submit the completed Devpost project.

# Acceptance Criteria

- [x] The authoritative demo documents describe the new story consistently.
- [x] Product documentation and provider context distinguish workspace-owned
      application auth/data/logs from Control Plane credentials and host state.
- [x] A rejected targeted server action writes a bounded error result to the
      generated page-state target and clears its pending state.
- [x] Every generated-product frame comes from a new public-server recording.
- [x] The first visible change turns the blank product into a dark theme.
- [x] The film visibly creates an account, signs out, and signs back in through
      generated product UI and server capabilities.
- [x] A timer/keyboard game works without refresh and the final product lists at
      least two games while preserving the signed-in account surface.
- [x] Visible prompt entry uses rapid sequential typing, Enter submission, and
      conversation/result scrolling rather than instant field fill.
- [x] Provider waits remain continuous but are accelerated enough that no single
      wait dominates the film; every accelerated segment has a speed label.
- [x] The contribution scene explicitly covers planning, implementation,
      presentation deck, film production, and the live Terra-medium server agent.
- [x] The final film is under 180 seconds with English narration and burned plus
      embedded English subtitles.
- [x] Public readiness, semantic runtime outcomes, media decode, subtitle timing,
      audio, and secret-safety checks pass.
- [x] The verified revision is pushed and deployed, its public canary passes,
      and the Devpost entry is submitted with the final video URL.
